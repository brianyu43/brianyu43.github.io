## 1. Recommended Korean blog-page sections

The current Korean page style works best as an editorial "systems story" rather than a literal lecture transcript. `podcast-ep-42.html` uses a strong hero/dek, a compact chapter strip, large act headings, callouts, metric/diagram modules, and short source links. For GPU Mode lecture pages, keep that rhythm but use lecture-grounded modules:

- **Hero**: Korean title, short dek, lecture number/date/speaker/source-confidence chips. Use the beige/rust visual system from `style.css`; avoid overloading the hero with every repo link.
- **Why this lecture matters**: one paragraph that places the lecture in the 001-026 progression: profiling -> CUDA basics -> memory/compute -> reductions/scan -> attention -> Triton/CUTLASS/NCCL -> vendor portability.
- **Chapters / acts**: 4-6 sections max. Good recurring act labels: "문제 설정", "하드웨어가 요구하는 모양", "커널/알고리즘의 핵심", "프로파일러가 보여주는 병목", "LLM 시스템과의 연결", "직접 해볼 실험".
- **Key mechanism card**: one visual card per page: launch timeline for 001/008/016, memory hierarchy for 004/012/018, reduction tree for 009/020/021/024, ring/all-reduce diagram for 013/017, Tensor Core tile for 015/023/025.
- **What to verify before publishing**: small QA box for caption-derived names/numbers, missing slides/code, and whether claims came from transcript vs repo material.
- **Hands-on path**: 2-3 concrete exercises using local repo materials when available. For source-thin lectures, say "slides/transcript review first" rather than inventing code.
- **Source footer**: YouTube, slides, code, notes path, caption status. This is important because the archive notes repeatedly warn that caption-derived names/acronyms/numbers need verification.

- **Global risk**: every note from 001-026 includes "Verify any caption-derived names, acronyms, and numeric claims against the original video/slides before relying on them." Do not publish exact performance numbers, people names, or quoted claims unless they are checked against slides/code/video.

Before writing titles/deks, prefer the lecture title, speaker, local code list, and core-idea section over generic "systems relevance" text.
- **Transcript noise**: captions are available for all 001-026, but transcript snippets contain recognition errors (`Trion`, `flesh attention`, `CLE/sickle`, etc.). Korean copy should normalize terminology only after checking the intended term.
- **Vendor specificity risk**: 025 and 026 are AMD ROCm/Composable Kernel and Intel SYCL pages. Avoid NVIDIA-only framing in these posts, even though some generated key-term lists still include CUDA/PTX/Tensor Core/FlashAttention.
- **Speculative decoding page risk**: 022 has a good abstract in local metadata but no local code material. Keep it conceptual and vLLM-architecture oriented unless additional code is added.
- **Hands-on profiling page risk**: 016 has no local code or slides inventory despite being a profiling lecture. A Korean blog page can still be useful, but should be framed as "profiling workflow notes from transcript" with a visible verification box.

## 4. High-signal title/dek examples

1. **`001` PyTorch에서 CUDA 커널까지, 첫 번째 병목은 어디에 찍히는가**  
   Dek: `torch` 호출, custom CUDA, Triton, Nsight 흐름을 한 페이지에 놓고 "느린 커널을 찾는 법"부터 시작한다.

2. **`004` GPU는 계산기보다 물류 시스템에 가깝다**  
   Dek: warp, shared memory, occupancy, tiling을 "데이터가 어디에 있고 얼마나 자주 움직이는가"라는 질문으로 다시 정리한다.

3. **`008` CUDA 성능 체크리스트: 빠른 코드는 운이 아니라 절차다**  
   Dek: coalescing, divergence, occupancy, shared memory 같은 체크 항목을 프로파일러 중심의 실전 점검표로 바꾼다.

4. **`009` Reduction은 덧셈이 아니라 교통정리다**  
   Dek: 같은 합계를 구해도 atomic, shared memory, warp-level 패턴에 따라 GPU가 완전히 다른 기계처럼 동작한다.

5. **`012` FlashAttention: attention을 저장하지 않고 흘려보내는 법**  
   Dek: softmax와 matmul 사이에서 HBM 왕복을 줄이는 아이디어를 tiled online computation 관점으로 설명한다.

6. **`014` Triton은 CUDA를 숨기지 않는다, 실험 속도를 바꾼다**  
   Dek: Python 안에서 커널을 쓰고 benchmark/autotune/debug 루프를 돌리는 실무 감각을 정리한다.

7. **`015` CUTLASS와 CuTe: 타일을 이해하면 라이브러리 코드가 보인다**  
   Dek: Tensor Core/MMA 커널을 직접 짜기 전에, CUTLASS가 어떤 모양으로 문제를 쪼개는지 읽는 법을 잡는다.

8. **`017` NCCL은 분산 학습의 보이지 않는 커널이다**  
   Dek: collective, reduction, DDP 예제를 통해 "GPU가 많아지면 계산보다 동기화가 먼저 보인다"는 감각을 만든다.

9. **`022` Speculative Decoding은 추측이 아니라 스케줄링 문제다**  
   Dek: vLLM의 proposer/scorer/verifier 구조를 중심으로, 빠른 생성이 커널 하나가 아니라 serving loop 전체의 문제임을 보여준다.

10. **`025-026` CUDA 밖의 GPU 프로그래밍: CK와 SYCL로 보는 이식성의 비용**  
    Dek: AMD Composable Kernel과 Intel SYCL 자료를 CUDA 관성에서 분리해, 같은 병목을 다른 vendor stack이 어떻게 표현하는지 비교한다.
