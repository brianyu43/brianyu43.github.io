## Recommended Korean blog-page sections

- **Hero / dek / metadata strip**: follow the `pod-hero` pattern from `podcast-ep-42.html`: a compact episode/lecture kick, one strong Korean title, one plain-language dek, then duration/speaker/source badges. This range has many tool/compiler talks, so the dek should answer "왜 지금 이걸 봐야 하나" before naming every acronym.
- **Source confidence box**: add a visible "자료 상태" block near the top. For 079-104, all local notes show `Transcript status: failed`; only 079, 084, 086, 103, and 104 have local slide PDFs. This should be a design feature, not a footnote.
- **What changes after watching this?**: a 3-bullet payoff section works better than a long intro. Suggested buckets: "커널을 보는 눈", "컴파일러/DSL 선택 기준", "서빙/분산 병목".
- **Chapter/act list**: reuse the chapter-list idiom from the podcast page, but label by concept rather than fake timestamps unless a verified transcript is available. Good act clusters for this range: mega-kernel/fusion, attention/CuTe/CUTLASS, DSL/compiler abstractions, communication/distributed inference, benchmarking/observability, ML engineering/RL/multimodal side quests.
- **Core mechanism card**: use `.callout`, `.kv`, `.tbl`, and `.viz` styles from `style.css` to show one mechanism per page: e.g. MPK fuses an inference path into one megakernel, FA4/CuTe maps layout to Blackwell kernels, NVSHMEM targets low-latency communication, InferenceX fights stale benchmark snapshots.
- **Risk / verification notes**: keep a short Korean "주의해서 읽을 것" section on every page. The archive itself repeatedly says the notes are partial and source-grounded rather than exhaustive.
- **Follow-up lab**: end with one practical exercise or profiling question. The generated notes often recommend small benchmarks, Nsight profiling, IR/PTX inspection, or reproducing an attention idea on small tensors.
- **Related lectures**: use cross-links sparingly. Strong local candidates: 080/086/103 around CuTe, 080/083 around FlashAttention, 079/096 around fusion/scheduling, 100/098 around benchmarking/observability, 101/103/104 around CUTLASS/CuTe/layouts.

## Content risks for lectures 079-104

- **Transcript risk across the whole range**: all 26 notes report failed transcript fetches. Do not write quote-style dialogue, timestamped claims, or "the speaker said" summaries unless manually checked against the video.
- **Sparse-source lectures**: 081, 085, 088, 089, 090, 092, 093, 094, 095, 097, 098, 099, 102, and 104 have weak extracted themes in local notes. Treat titles/speakers/descriptions as safe, but avoid detailed technical claims without slide/video review.
- **Slides are uneven**: local official repo materials were found only for 079 `Mirage-2025-09-13.pdf`, 084 `numerics-and-ai.pdf`, 086 `cute_dsl_introduce.pdf`, 103 `layout_algebra.pdf`, and 104 `gluon.pdf`.
- **Speaker metadata mismatch risk**: 086 appears as "Speakers: ..." in YouTube description but the generated note names Vicki Wang. 083 and 091 show missing speaker in notes despite descriptions containing useful context. Verify names before publishing.
- **Acronym-heavy pages can become opaque in Korean**: Mirage/MPK, FA4, CuTe DSL, NVSHMEM, cuTile, tvm-ffi, TLX, HipKittens, Monarch, Gluon, and quartet need first-use expansion or a one-sentence "이게 뭔가" line.
- **Benchmark claims need source labels**: 079 includes a local metadata claim of 1.2x-6.7x latency reduction for Mirage MPK; 100 describes continuous OSS inference benchmarking. Keep numbers tied to their source and avoid generalizing to all workloads.
- **Current design can support the range, but avoid overusing fictional podcast framing**: `podcast-ep-42.html` is explicitly marked `IMAGINARY` and uses Sam/Jensen dialogue. For lecture pages, prefer analyst notes, mechanism cards, and diagrams unless the page is intentionally labeled as fiction.
- **Mobile design watchpoint**: the CSS has responsive handling for hosts/two-column acts and smaller titles, but chapter rows use fixed-ish grid columns. Long Korean titles plus acronyms may wrap awkwardly; keep chapter labels short.

## High-signal title/dek examples

- **079 Mirage MPK: LLM 추론을 하나의 메가커널로 접는 법**  
  Dek: 커널 런치와 메모리 왕복을 줄이기 위해 Mirage가 필요한 계산과 통신을 한 번의 persistent kernel로 묶는다는 아이디어를, "fusion이 어디까지 갈 수 있나"라는 질문으로 읽는다.

- **080 FlashAttention 4: Blackwell에서 attention 커널은 무엇을 새로 배웠나**  
  Dek: 공개된 FA4/CuTe 구현과 reverse-engineering 글을 출발점으로, FlashAttention이 버전마다 HBM traffic, scheduling, layout 문제를 어떻게 다시 풀어왔는지 정리한다.

- **083 Formalized Kernel Derivation: FlashAttention을 손그림이 아니라 수식에서 끌어내기**  
  Dek: category-theoretic diagram과 resource model로 딥러닝 연산에서 low-level kernel을 유도하려는 시도. 단, 현재 로컬 자료는 설명문 중심이라 알고리즘 세부는 검증이 필요하다.

- **086 CuTe DSL 입문: 타일, 레이아웃, Tensor Core를 한 언어로 말하기**  
  Dek: Blackwell/NVFP4 흐름에 맞춰 CuTe DSL이 어떤 추상화로 커널 작성을 돕는지 보고, CUTLASS 계열 코드가 어려운 이유를 레이아웃 언어 관점에서 풀어낸다.

- **087 NVSHMEM 통신 커널: decode-heavy LLM에서 네트워크가 병목이 되는 순간**  
  Dek: tensor parallel inference가 커질수록 계산보다 통신이 앞에 선다. NVSHMEM을 low-latency communication kernel의 관점에서 읽고, 같은 노드/다른 노드의 비용 감각을 세운다.

- **096 TLX: Triton처럼 쓰되 Blackwell 스케줄링을 놓치지 않는 DSL**  
  Dek: TLX는 Triton-like 모델에서 mechanical complexity를 덜어내면서 warp specialization, async pipeline, memory-system control 같은 저수준 선택지는 남기려는 시도다.

- **100 InferenceX: LLM 추론 벤치마크가 낡는 속도와 싸우기**  
  Dek: vLLM, SGLang, TensorRT-LLM 최적화가 며칠 단위로 바뀌는 환경에서 point-in-time benchmark가 왜 빨리 부패하는지, continuous OSS benchmarking으로 본다.

- **101 CUTLASS를 어렵게 배우기: GEMM 커널의 추상화 밑바닥으로 내려가기**  
  Dek: CUTLASS는 빠른 라이브러리이기 전에 GPU matrix instruction, tiling, layout, pipeline을 노출하는 언어다. "어렵다"는 감각 자체를 학습 경로로 삼는다.

- **103 CuTe Layout Algebra: 커널 성능을 좌표계 문제로 다시 보기**  
  Dek: CuTe의 layout algebra와 category-theoretic 해석을 통해, 메모리 배치와 thread/value mapping이 단순한 구현 디테일이 아니라 커널 설계의 문법임을 보여준다.

- **104 Gluon and Linear Layouts: 다음 커널 DSL은 레이아웃을 어떻게 말할까**  
  Dek: Gluon과 linear layouts를 중심으로 최신 GPU kernel DSL이 레이아웃 표현을 어디까지 밀어붙이는지 살핀다. 로컬에는 슬라이드가 있으나 transcript가 없어 세부 주장은 검증 후 확장해야 한다.
