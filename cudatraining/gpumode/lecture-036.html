03 · TERMS용어와 맥락

이 강의를 읽기 위한 작은 사전.

Glossary18 entries
warp — A group of GPU threads that execute in lockstep on NVIDIA GPUs.
kernel — A GPU program launched over many threads/blocks to process data in parallel.
MMA — Matrix multiply-accumulate instruction family used by Tensor Cores.
softmax — Exponentiate-normalize operation, commonly fused in attention kernels.
shared memory — Program-managed on-chip SRAM used for tiling and reuse within a block.
FlashAttention — An attention algorithm/kernel family that reduces HBM traffic with online softmax and tiling.
TMA — Tensor Memory Accelerator, hardware support for bulk tensor movement on Hopper-class GPUs.
scheduling — Choosing execution order, tiling, parallelism, and memory movement strategy.
CTA — Cooperative Thread Array, commonly called a thread block in CUDA.
CuTe — CUTLASS layout/tensor abstraction library used to express tiled GPU kernels.
collective — Distributed communication operation such as all-reduce, broadcast, or all-gather.
CUDA — NVIDIA's GPU programming platform and kernel programming model.
HBM — High Bandwidth Memory, the off-chip memory backing large tensor data.
tiling — Splitting work/data into blocks that fit cache/shared memory/registers.
normalization — Operations such as LayerNorm/RMSNorm that rescale activations.
SM — Streaming Multiprocessor, the GPU execution unit that schedules warps.
Tensor Core — Matrix-multiply hardware units optimized for mixed-precision GEMM-like workloads.
PTX — NVIDIA's virtual GPU ISA used as an intermediate target before SASS.

PROF
Start performance work by separating memory bandwidth limits from compute limits; use profiler counters rather than wall-clock timing alone.
KERNEL
For matmul-like kernels, tile for data reuse, align work with Tensor Core/MMA shapes when available, and validate against vendor baselines.
PROF
For attention kernels, optimize HBM traffic first: stream blocks, keep online softmax state in registers/SRAM, and avoid materializing large score matrices.
KERNEL
In Triton, benchmark multiple block sizes/num-warps/stages; good source code can still be slow under the wrong launch configuration.
PROF
For multi-GPU work, model communication overlap and collective latency; scaling failures often come from synchronization and network topology, not only kernels.
